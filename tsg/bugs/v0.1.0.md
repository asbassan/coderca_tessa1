# Tessa1 Bug Log - v0.1.0

This file captures the recent bugs and runtime issues found while hardening Tessa1 for CLI usage and container packaging.

## 1. GitHub Copilot SDK false-negative during `init`

**Symptom**

`coderca_tessa1 init` reported:

```text
GitHub Copilot SDK not available: cannot import name 'MessageOptions' from 'copilot'
```

even though `github-copilot-sdk` had been installed successfully in the container image.

**Root cause**

Tessa1's Copilot adapter expected one exact SDK import layout:

- `from copilot import MessageOptions`

But the installed SDK surface can vary by version/layout, and `MessageOptions` may need to be resolved from:

- `copilot.types`

Because the adapter treated that import mismatch as "SDK unavailable", `init` failed the default real-LLM readiness check incorrectly.

**Fix**

- made the adapter try both import paths
- kept the SDK marked available if either import path succeeds
- exposed the real import error only when all import attempts fail

**Status**

- fixed

---

## 2. Old CLI surface was still CodeRCA-shaped and broken

**Symptom**

The copied CLI still contained old RCA/scenario commands and duplicate scenario registration. That made the public CLI surface confusing and, in some cases, import-time fragile/broken.

**Root cause**

Tessa1 was still carrying a copied `cli.py` from CodeRCA instead of a Tessa-first command surface.

**Fix**

- replaced the old CLI with a Tessa-first CLI
- added:
  - `coderca_tessa1 init`
  - `coderca_tessa1 --init`
  - `coderca_tessa1 list-users`
  - `coderca_tessa1 demo-feed`
- removed the copied RCA/scenario path from the public interface

**Status**

- fixed

---

## 3. Feed report formatting collapsed sections together

**Symptom**

The rendered report had sections running together, for example:

- `Candidates evaluated: 20Overall Summary:`
- ranked post blocks with no clean separation
- `Generated Explanations:` and `Architecture Mapping:` not clearly separated

**Root cause**

`FeedReport.to_text()` did not insert enough blank-line boundaries between sections and post blocks.

**Fix**

- added clean section spacing
- added separation between ranked post blocks
- added tests to lock in multi-post formatting behavior

**Status**

- fixed

---

## 4. `SynthesisAgent` was under-agentified

**Symptom**

The synthesis phase was mostly sentence formatting and did not yet behave like a real explanation-planning stage.

**Root cause**

The original implementation only echoed deterministic score facts instead of:

- selecting salient reasons
- adapting explanation style
- comparing adjacent ranked posts

**Fix**

- added explanation planning
- added audience-style selection from profile modes
- added salient-fact selection
- added contrastive notes between adjacent posts
- added focused synthesis tests

**Status**

- fixed

---

## 5. `init` check order did not match the intended setup flow

**Symptom**

The init flow initially checked files before:

1. checking GitHub Copilot SDK availability
2. creating `artifacts` and `runlogs`

That made the setup flow harder to explain and did not match the intended operator sequence.

**Root cause**

The first implementation of `init` grouped file checks before readiness/setup checks.

**Fix**

The init flow now runs in this order:

1. check GitHub Copilot SDK availability
2. create `artifacts` and `runlogs`
3. verify:
   - `data/users.json`
   - `config/scoring.json`
   - `data/raw/uci_news/News_Final.csv`

**Status**

- fixed

---

## 6. Tessa1 had no explicit init/preflight path

**Symptom**

Users could run the CLI, but there was no dedicated command to validate:

- required files
- writable output paths
- default real-LLM prerequisites

**Root cause**

Tessa1 originally had only the runtime commands, not a setup validation command.

**Fix**

- added `coderca_tessa1 init`
- added `coderca_tessa1 --init`
- added tests for successful init, failure cases, and global init flag behavior

**Status**

- fixed

---

## 7. Default runtime behavior did not match deployment expectations

**Symptom**

The earlier CLI behavior required more explicit flags than desired for container/runtime use.

Examples:

- real LLM mode was not the default path
- report artifacts were not automatically produced by default

**Root cause**

The earlier CLI defaults were optimized for manual demo runs rather than deployment-oriented execution.

**Fix**

- made real LLM mode the default CLI path
- added default report artifact output under `artifacts\`
- kept opt-out switches:
  - `--no-use-real-llm`
  - `--no-save-report`
  - `--no-save-runlog`

**Status**

- fixed
